# Emailer (subfolder of `arams-db`)

Apps Script web app that sends Gmail messages on behalf of Aram's Das Experten
ecosystem. It supports three call patterns:

- **A** — new email with a pre-made attachment link (no Doc generation)
- **B** — new email whose body and attached Google Doc are generated by a
  Das Experten skill (loaded from `../.claude/skills/`)
- **C** — reply inside an existing Gmail thread, preserving threading headers

This is **not a standalone repository** — it is a subfolder of
[`dasexperten/arams-db`](https://github.com/dasexperten/arams-db). The skills
knowledge base lives next to it under `arams-db/.claude/skills/` and is loaded
into Apps Script `PropertiesService` at deploy time (Apps Script cannot read
local files at runtime).

```
    │   ├── EmailComposer.gs       (subject + HTML/plain bodies)
    │   ├── DocBuilder.gs          (Google Doc generation)
    │   ├── DriveManager.gs        (Drive folder + share links)
    │   ├── GmailSender.gs         (sendNew + replyToThread)
    │   ├── ThreadResolver.gs      (Gmail thread context)
    │   ├── SkillsBridge.gs        (PropertiesService-backed skill loader)
    │   └── Logger.gs              (Sheet logging)

```

External callers (n8n, Make, the dasoperator ERP, Aram's terminal) hit the
deployed web-app URL with a JSON payload. The web app routes the request,
optionally renders content via a skill template, builds and shares a Doc when
needed, and sends Gmail.

```
[ caller ] --HTTPS POST JSON--> [ Apps Script web app: doPost ]
                                       │
                                       ├── ThreadResolver  (if reply)
                                       ├── SkillsBridge    (if skill_call)
                                       ├── DocBuilder      (if no attachment)
                                       ├── DriveManager    (if no attachment)
                                       ├── EmailComposer   (always)
                                       ├── GmailSender     (always)
                                       └── Logger          (always)
```

## Relationship to `.claude/skills/`

Skills are markdown templates living one folder up at
`arams-db/.claude/skills/<skill-name>/SKILL.md`. Apps Script cannot read the
local repo at runtime, so the deploy procedure (see `SETUP_NOTES.md`) loads
each `SKILL.md` into a `PropertiesService` key named `skill_<skill-name>`.

At runtime, `SkillsBridge.callSkill(name, payload)`:

1. Loads `skill_<name>` from `PropertiesService`.
2. Renders `{{key}}` placeholders with values from `payload`.
3. Returns `{ content, metadata }` for `EmailComposer` and `DocBuilder`.

If a skill is requested but its template is not yet deployed, `SkillsBridge`
returns a clearly-marked stub so the rest of the pipeline still runs.

## Call format

`POST <web app URL>`, content type `application/json`.

### Schema

```json
{
  "task": "string (required, human-readable label)",
  "recipient": "string (email, required for new emails)",
  "subject": "string (optional, ignored on reply)",
  "content_brief": "string (required, what to write about)",
  "attachment_link": "string (optional, pre-made artifact URL)",
  "skill_call": "string (optional, skill name from .claude/skills/)",
  "context": "string (optional, extra context)",
  "thread_id": "string (optional, Gmail thread ID — triggers reply mode)",
  "in_reply_to_message_id": "string (optional)"
}
```

### Response

```json
{
  "success": true,
  "mode": "new | reply",
  "message_id": "Gmail message ID",
  "thread_id": "Gmail thread ID",
  "doc_link": "Drive share URL or null",
  "log_id": "row number in log sheet or null",
  "error": null
}
```

### Scenario A — new email with pre-made attachment

```json
{
  "task": "Send Q2 distributor deck to Vietnam buyer",
  "recipient": "buyer@vn-distrib.example",
  "subject": "Das Experten — Q2 distributor deck",
  "content_brief": "Cover note with the attached distributor deck.",
  "attachment_link": "https://drive.google.com/file/d/EXISTING_ID/view",
  "context": "Vietnam, pharmacy chain, follow-up to last week's call."
}
```

`DocBuilder` and `DriveManager` are skipped — `attachment_link` goes straight
into the email.

### Scenario B — new email, content + Doc generated by a skill

```json
{
  "task": "Brief on SCHWARZ paste for Polish wholesaler",
  "recipient": "lead@pl-wholesale.example",
  "content_brief": "Explain SCHWARZ positioning, RDA, and target shopper.",
  "skill_call": "product-skill",
  "context": "Wholesaler asked about whitening line; price tier mid."
}
```

`SkillsBridge` renders the `product-skill` template, `DocBuilder` builds a
Google Doc, `DriveManager` shares it, `EmailComposer` writes the body, and
the email goes out.

### Scenario C — reply inside an existing Gmail thread

```json
{
  "task": "Reply to Vietnam buyer about MOQ",
  "thread_id": "186b7a8c1234abcd",
  "content_brief": "Confirm MOQ 1 pallet, lead time 4 weeks, FOB Riga.",
  "context": "Buyer pushed back on MOQ in last reply."
}
```

`ThreadResolver` validates access and pulls thread context (subject, last
sender, snippet), `EmailComposer` writes a reply that acknowledges the prior
message, and `GmailSender.replyToThread` posts inside the same Gmail thread —
Gmail handles `In-Reply-To` and `References` headers automatically, so no
orphan messages are produced.

## Setup

See [`SETUP_NOTES.md`](./SETUP_NOTES.md) for the full clasp-based deploy
procedure. High level:

1. `cd arams-db/emailer && clasp create --type webapp --title "Emailer"`
2. `clasp push` to upload all `src/*.gs` files and `appsscript.json`.
3. Create a Google Sheet for logging, copy its ID.
4. Deploy as web app (execute as me, anyone with link).
5. Set `PropertiesService` keys: `LOG_SHEET_ID`, `DEFAULT_DRIVE_FOLDER`.
6. Load each `../.claude/skills/<name>/SKILL.md` into `skill_<name>` via clasp.
7. Smoke-test the deployed URL with the three sample payloads above.

## Limits and caveats

- Apps Script cannot read sibling files in this repo at runtime — skills must
  be deployed into `PropertiesService` (the loader is a one-liner described in
  `SETUP_NOTES.md`).
- Gmail daily quota: 1,500 messages/day for Workspace, 100/day for consumer
  accounts. Watch for `Service invoked too many times` errors.
- `GmailApp.sendEmail` does not return IDs directly; `sendNew` re-queries the
  Sent mailbox to look up the freshly-sent message — works in practice, but
  expect one search query per send.
- All credentials live in `PropertiesService` — no secrets in code, no
  secrets in this repo.
